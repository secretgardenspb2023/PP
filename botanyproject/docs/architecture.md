# Архитектура PoiskPlant

## Обзор

Монорепозиторий: Django-backend (DRF API) + Next.js-frontend, общая инфраструктура
в Docker Compose. Поиск вынесен в Elasticsearch, медиа — в S3 с обработкой через
imgproxy, кэш и брокер задач — Redis + Celery.

```
            ┌─────────┐     ┌──────────────┐
 браузер ──▶│  Nginx  │────▶│ Next.js (SSR)│
            │ (proxy) │     └──────────────┘
            │         │     ┌──────────────┐     ┌────────────┐
            │         │────▶│ Django + DRF │────▶│ PostgreSQL │
            │         │     │  (gunicorn)  │     └────────────┘
            │         │     │              │────▶┌────────────┐
            │         │     │              │     │   Redis    │
            │         │     │              │────▶┌────────────┐
            │         │     └──────┬───────┘     │Elasticsearch│
            │  /img/  │            │             └────────────┘
            │ (cache) │────▶ imgproxy ──▶ S3 (Yandex Object Storage)
            └─────────┘     Celery worker ◀── Redis (broker)
```

## Многосхемная БД

Существующая база клиента использует несколько PostgreSQL-схем:

- `plant_info` — характеристики растений (есть в дампе);
- `plant` — основная таблица `plants` (название, slug, статус, фото) — **дампа пока нет**;
- `user_info` — пользователи (`user_info.users` — основная таблица User по решению заказчика);
- `django_internal` — служебные таблицы Django (пересоздаётся).

Подробности и план нормализации — [database.md](database.md).
