# Техническое задание и план работ

## Электронный справочник растений PoiskPlant

**Версия: 2.2   Дата: 5 июня 2026 г.**

Документ фиксирует согласованный объём работ, сроки, этапы оплаты, требования к заказчику и протокол совместной работы. Версия 2.2 учитывает все правки заказчика: использование `user_info.users` как основной таблицы пользователей, бесплатный email через Yandex Mail для домена (отказ от Unisender), подключение Яндекс.Метрики с cookie-баннером. Подтверждение документа = старт работ.

---

## 1. Описание проекта

Разработка нового электронного справочника растений PoiskPlant на современном стеке. Существующий сайт работает на WordPress/WooCommerce и продолжит работу до релиза новой версии; контент уже перенесён заказчиком в самостоятельную PostgreSQL-базу (~12 371 карточка).

Стек новой версии: Django (backend) + Next.js (frontend) + PostgreSQL + Elasticsearch + S3 + imgproxy. Развёртывание в Docker на VPS HandyHost. Продвинутый поиск с морфологией русского языка и фасеточными фильтрами, расширенная админка для работы с базой, авторизация через 4 способа (Email + Google + VK + Telegram). На текущем этапе создание карточек закрытое — только администратор и редакторы. Открытое создание карточек, избранное, отзывы и жалобы — Фаза 2.

## 2. Состав работ по этапам

### Этап 1. Инфраструктура, безопасность, интеграция с существующей БД

1. Подготовка VPS HandyHost: Ubuntu 24.04, sudo-пользователь, SSH по ключу, отключение root и парольного входа, нестандартный SSH-порт, автообновления безопасности.
2. Firewall (UFW): только SSH/80/443. Fail2ban для SSH, Nginx и DRF auth-эндпоинтов.
3. Docker + Docker Compose, изоляция контейнеров.
4. Nginx как reverse proxy с заголовками безопасности (HSTS, CSP, X-Frame-Options и др.), сжатием, кэшем статики.
5. SSL-сертификат Let's Encrypt с автообновлением.
6. PostgreSQL в контейнере с настройкой под объём (~12 371 карточка), раздельными ролями (app-user CRUD, backup-user readonly).
7. Redis в контейнере (кэш, брокер Celery, rate-limit).
8. Elasticsearch в контейнере (для продвинутого поиска с морфологией русского).
9. Анализ существующей Postgres-схемы заказчика: документирование структуры, выявление избыточностей и недостающих полей, предложение по доработкам (нормализация связей, индексы под Elasticsearch, добавление полей для нового функционала). Все изменения схемы согласуются с заказчиком до применения.
10. Бэкапы: ежедневный `pg_dump` → шифрованная выгрузка в S3; ротация 7 ежедневных + 4 еженедельных + 12 месячных; ежемесячное автоматическое тестовое восстановление.
11. Мониторинг: ротация Docker-логов, UptimeRobot, Sentry для error-tracking, алерты в Telegram.
12. Staging-окружение на поддомене `staging.домен` с HTTP Basic Auth и закрытое от индексации.
13. CI/CD: GitHub Actions с линтерами и тестами перед деплоем.

### Этап 2. Каркас backend

1. Django 5.x проект, разделение настроек (dev/staging/prod), `django-environ`.
2. Зависимости: DRF, `django-cors-headers`, `django-storages`, `celery`, `psycopg`, `gunicorn`.
3. Кастомная модель User с email в роли USERNAME_FIELD.
4. Подключение к PostgreSQL, проектирование моделей Django под существующую и расширенную схему данных.
5. DRF: пагинация, throttling, версионирование `/api/v1/`.
6. CORS whitelist, JSON-логирование для Docker.
7. Gunicorn: 2×CPU+1 воркеров, `--max-requests`.
8. Health-check `/api/v1/health/` с проверкой БД, Redis, S3, Elasticsearch.

### Этап 3. Авторизация

1. **Кастомная модель User в Django, привязанная к существующей таблице `user_info.users`** (основная таблица пользователей по решению заказчика). Старая таблица `django_internal.auth_user` пересоздаётся.
2. Создание административного аккаунта заказчика в `user_info.users` с правами `is_superuser` (логин, email и пароль предоставляет заказчик).
3. Email-регистрация с подтверждением, сбросом пароля, валидацией.
4. Стратегия сессий: refresh в HttpOnly cookie + access-токен.
5. `django-allauth` + 3 OAuth: Google, VK ID (кастомный адаптер), Telegram (Login Widget с проверкой подписи). Значение MAX в `social_provider` оставлено в схеме «на будущее» (публичный OAuth для VK MAX пока недоступен).
6. Безопасное объединение аккаунтов по email — только при подтверждённом email от провайдера.
7. Защита от подбора пароля: `django-axes` с экспоненциальным backoff.
8. Yandex SmartCaptcha на регистрации, сбросе пароля, входе после неудачной попытки.
9. CSRF для cookie-based флоу.
10. Логирование auth-событий: вход, выход, смены пароля, привязки.
11. Профиль: редактирование, список привязанных соц-сетей с отвязкой, активные сессии.
12. 2FA (TOTP) для администраторов и модераторов — `django-otp` + приложение-аутентификатор (Google Authenticator, Aegis и т.п.).

### Этап 4. Медиа-система

1. S3-bucket уже создан заказчиком (Yandex Object Storage), интеграция через `django-storages`.
2. Модель PlantImage (FK, s3_key, uploaded_by, is_primary, alt_text).
3. Скрипт миграции фотографий: фото хранятся в существующей базе как внешние ссылки → скачивание по ссылкам через Celery (с защитой от SSRF, таймаутами, контролем размера) → загрузка в S3 → обновление записей в БД с привязкой к карточкам. Скрипт идемпотентный (повторный запуск не дублирует уже мигрированные фото).
4. Загрузка фото из админки: валидация mimetype/размера/формата через Pillow.
5. imgproxy: HMAC-подписанные URL, доступ к S3.
6. Пресеты: `thumbnail` (200px), `card` (400px), `full` (1200px), `hero` (2000px). WebP/AVIF по Accept-заголовку браузера.
7. Кэширование `/img/` через Nginx proxy_cache.
8. Management-команда очистки осиротевших файлов с dry-run.

### Этап 5. API, продвинутый поиск, фильтры

1. Модели: Plant + категории, характеристики, теги, фото, регионы, период цветения, условия содержания.
2. CRUD-эндпоинты: список с пагинацией/фильтрами/сортировкой, детальная карточка, создание (только админ/редактор), редактирование, фильтры с фасеточными счётчиками, загрузка фото.
3. Elasticsearch: индекс по карточкам, синхронизация через `django-elasticsearch-dsl` и Celery.
4. Морфологический анализ русского языка (поиск «посадка» находит «посадить», «посаженный»).
5. Fuzzy-поиск с автоматическим подбором расстояния Левенштейна.
6. Словарь синонимов: научные ↔ народные названия, латинские ↔ русские (формат и содержание согласуются с заказчиком).
7. Edge ngram для мгновенных автоподсказок.
8. Ранжирование с весами полей (название важнее описания), boost популярных карточек.
9. Подсветка совпадений в результатах (highlight).
10. Эндпоинт автоподсказок `/api/v1/search/suggest/` с откликом <50 мс.
11. Fallback на `pg_trgm` при недоступности ES.
12. Фасеточные фильтры со счётчиками (как в маркетплейсах).
13. Мульти-выбор внутри категории (OR), пересечение между категориями (AND).
14. Диапазонные фильтры (высота, диаметр, температура, влажность) с гистограммами распределения.
15. Зависимые фильтры (при выборе категории — специфичные для неё характеристики).
16. Сохранение фильтров в URL (shareable links).
17. Сортировка: по релевантности, алфавиту, новизне, популярности.
18. Swagger через `drf-spectacular` на `/api/v1/schema/`.
19. Тесты (`pytest-django`, `factory_boy`): happy-path и негативные сценарии по каждому эндпоинту.
20. Redis-кэш тяжёлых запросов с инвалидацией при изменении данных.

5. Canonical URL.

## 5. Протокол совместной работы с базой данных

Заказчик имеет существующую PostgreSQL-базу с контентом (~12 371 карточка) и продолжает работу с ней (наполнение, правки) на протяжении проекта. Чтобы не блокировать заказчика и одновременно безопасно вести разработку, применяется следующий протокол:

### Принципы

- Заказчик в любой момент видит полную документацию структуры БД (актуальную). Документация ведётся в git вместе с кодом моделей Django и обновляется при каждом изменении схемы.
- Все изменения схемы БД (новые таблицы, поля, индексы, переименования) — только по согласованию с заказчиком. Самовольные структурные правки исключены.
- Все изменения схемы делаются через миграции Django, отслеживаются в git, обратимы.
- Параллельная работа: у заказчика — рабочая база, у исполнителя — staging-копия. Прямая работа с боевой базой исполнителем исключена до момента финального переключения.

### Регламент

1. **Старт:** заказчик передаёт текущий дамп базы. Исполнитель анализирует структуру, предлагает рекомендации по доработке схемы (документ с обоснованием каждого изменения).
2. **Согласование:** заказчик утверждает изменения, либо предлагает корректировки. Финальная схема фиксируется.
3. **Разработка:** исполнитель работает на staging-копии. Заказчик продолжает наполнение/правки на рабочей базе.
4. **Синхронизация** (раз в 1–2 недели или по запросу): свежий дамп с рабочей базы → накатывается на staging → проверяется совместимость с новой схемой → заказчик видит свои актуальные данные в новом интерфейсе.
5. **Изменения схемы по ходу разработки:** исполнитель предлагает изменение → согласование с заказчиком → миграция применяется на staging → отражается в документации.
6. **Финальный переход** (платёж 3): заказчик за 1–2 часа до переключения прекращает правки → финальный дамп → миграция в новую схему → переключение DNS → возобновление работы на новой базе.

Документация структуры БД готовится в формате README в репозитории и обновляется по каждому изменению. Заказчик имеет к ней доступ с первого дня и может комментировать предлагаемые изменения.

## 9. Стратегия кэширования

Кэширование реализовано на нескольких уровнях, чтобы каталог из 12 371 карточки работал быстро при любой нагрузке.

**Уровни:**

- **Браузер:** статика (JS, CSS, шрифты) — Cache-Control до года с версионированием по контенту; обработанные картинки imgproxy — на месяц; HTML — короткий кэш для гостей, без кэша для авторизованных.
- **Nginx (reverse proxy):** proxy_cache на диске для картинок imgproxy (это критично — без него CPU будет уходить только на пересжатие фото); кэш статики Next.js; короткий кэш некоторых API-эндпоинтов (например, список доступных значений фильтров).
- **Redis:** тяжёлые SQL-запросы (списки растений с фильтрами и фасеточными счётчиками, TTL 5–15 минут с инвалидацией при изменениях); словари значений фильтров (TTL час); топ популярных карточек и поисковых запросов (TTL 10 минут); rate-limit счётчики; сессии Django.
- **PostgreSQL:** shared_buffers (~1.5 GB) для горячих страниц; pg_stat_statements для отслеживания медленных запросов.
- **Elasticsearch:** filter cache для фасеточных счётчиков (после первого выполнения запрос отвечает мгновенно); request cache для повторяющихся поисков.

**Ожидаемое поведение:**

- Первый запрос каталога с фильтрами: 50–100 мс.
- Повторный запрос в течение 5 минут (другой пользователь): 1–5 мс (Redis).
- Картинка карточки в любом размере после первой генерации: мгновенно (Nginx).
- Поиск с автоподсказками: <50 мс (Elasticsearch + Redis).
- Если данные карточки изменились — соответствующие кэши инвалидируются автоматически.

CDN (Cloudflare / BunnyCDN) на текущем этапе не подключается — производительности Nginx-кэша достаточно. При значительном росте трафика подключение CDN — отдельная задача в рамках поддержки, не требует переделок архитектуры.

## 10. Обоснование требований к серверу

Минимально рекомендуемая конфигурация: **4 vCPU / 8 GB RAM**. Минимально допустимая: 4 vCPU / 6 GB RAM. Меньше 6 GB — не рекомендуется категорически.

**Распределение памяти по сервисам:**

- Elasticsearch — 2 GB heap минимум + 1–2 GB OS file cache для индексов (с морфологией русского работает заметно тяжелее);
- PostgreSQL — 1–1.5 GB (shared_buffers, work_mem под несколько коннектов);
- Redis — 300–500 MB;
- Django + Gunicorn (4 воркера × ~400 MB) — около 1.5 GB;
- Next.js (SSR) — 500–800 MB;
- imgproxy, Nginx, Celery воркеры — суммарно около 500 MB;
- Ubuntu + Docker-оверхед — около 1 GB.

Суммарно минимум 6.5–7 GB активной памяти + запас на пиковую нагрузку. По линейке HandyHost подходит тариф с 8 GB RAM или ближайший выше. Заказчик выбирает тариф самостоятельно с учётом этих требований.
